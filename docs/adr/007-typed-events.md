# ADR-007: NatsEvents interface augmentation for typed subjects

**Status:** Accepted  
**Date:** 2026-05-11

## Context

`jsPublish` accepts a subject string and a payload. Without typing, the payload is `unknown` and subject strings are unchecked. This means:

- Typos in subject names are silent runtime errors
- Payload shape mismatches are only caught at runtime
- No IDE autocomplete for subjects or payloads

TypeScript module augmentation allows third-party types to be extended by user code without modifying the library source.

## Decision

Export an empty `NatsEvents` interface from the module:

```ts
// src/runtime/server/utils/publish.ts
export interface NatsEvents {}
```

Users augment it in their application:

```ts
// types/nats.d.ts
declare module 'nuxt-nats' {
  interface NatsEvents {
    'orders.created': { id: string; total: number }
    'user.registered': { id: string; email: string }
  }
}
```

`jsPublish` has two overloaded signatures:

```ts
// Typed path — subject must be a key of NatsEvents, payload must match
export async function jsPublish<S extends keyof NatsEvents>(
  subject: S,
  payload: NatsEvents[S],
  opts?: PublishOpts,
): Promise<void>

// Untyped fallback — any subject string, any JSON-compatible payload
export async function jsPublish(
  subject: string,
  payload: AnyPayload,
  opts?: PublishOpts,
): Promise<void>
```

TypeScript picks the typed overload when the subject matches a key in `NatsEvents`, and falls back to the untyped overload for unknown subjects.

## Consequences

- **Incremental adoption.** Teams can add entries to `NatsEvents` gradually. Untyped publishes still work; they just don't benefit from type checking.
- **Single source of truth for event contracts.** `NatsEvents` in `types/nats.d.ts` documents the entire event vocabulary of the application. It can be committed and reviewed like any other interface.
- **Works across the server directory.** Because `jsPublish` is auto-imported, the types flow everywhere in `server/` without explicit imports.
- **Does not enforce consumer payload types.** The `handler` in `defineNatsConsumer` is typed via the generic `T` parameter separately. A future improvement could derive `T` from `NatsEvents[subject]` automatically.
- **No runtime validation.** The types are compile-time only. If a producer sends a payload that doesn't match the declared shape (e.g., from a non-TypeScript service), `JSON.parse` will succeed and the type assertion will be incorrect. Users who need runtime validation should apply Zod or Valibot in their handlers.

## Alternatives considered

**String literal union type for subjects:** `type NatsSubject = 'orders.created' | 'user.registered'`. Less powerful — no payload association per subject. Rejected.

**Separate type registry file generated by the module:** Too complex, requires code generation infrastructure. Augmentation achieves the same result with zero tooling. Rejected.

**Zod schema registry built into the module:** Adds a required dependency (Zod) and runtime overhead for every publish. Rejected — validation is the user's responsibility at system boundaries.
